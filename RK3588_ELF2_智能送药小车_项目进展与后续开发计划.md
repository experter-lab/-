# RK3588 ELF 2 智能送药小车项目进展与后续开发计划

本文记录当前智能送药小车项目已经完成的内容、当前做到的位置、后续还需要完成的工作，以及从 PC 端开发迁移到 RK3588 的推荐路线。

当前明确原则：

```text
先在电脑端完成业务流程和软件原型
尽早在 RK3588 上做 Docker 部署和硬件冒烟测试
后续再逐步接入真实底盘、雷达、摄像头、语音和导航
```

---

## 1. 项目总体目标

本项目计划基于 **RK3588 ELF 2 开发板** 实现智能送药小车，面向医院病区、养老院、社区卫生服务中心等场景，实现药品配送、任务管理、语音交互、网页监控、药物信息识别和后续自主导航。

目标系统能力：

```text
网页下发送药任务
语音下发送药任务
药物信息识别与装药确认
任务状态实时显示
自动导航到指定病房或站点
到站提醒和任务播报
异常状态告警
后续支持远程管理和真实机器人部署
```

## 6. 摄像头模块当前定位

当前摄像头作用先确定为：

```text
获取药物信息
```

当前阶段不把摄像头作为主要避障或行人检测模块，先聚焦药物相关业务。

### 6.0 实际摄像头型号和关键参数

当前项目使用的摄像头为：

```text
DECXIN-2923V1 USB 摄像模组
```

关键参数：

```text
接口类型：Type-C USB3.0 24P 立式插座
Linux 兼容：支持 UVC，通常识别为 /dev/videoX
供电电压：USB 5V
工作电流：180 mA - 200 mA
感光芯片：1/2.0 英寸
最高有效像素：1280 × 1024
快门类型：Global Shutter 全局快门
输出格式：MJPEG / YUV2(YUYV)
对焦方式：固定对焦
镜头焦距：2.8 mm
光圈：F2.4
视场角：118° 对角
模组尺寸：38 mm × 38 mm × 24 mm
工作温度：0°C ~ 70°C
```

分辨率和帧率能力：

```text
MJPEG:
1280×1024 @ 200 fps
1280×720  @ 200 fps
960×540   @ 200 fps
848×480   @ 200 fps
640×512   @ 400 fps
640×480   @ 400 fps

YUYV:
1280×1024 @ 100 fps
1280×720  @ 100 fps
960×540   @ 200 fps
848×480   @ 200 fps
640×512   @ 400 fps
640×480   @ 400 fps
```

对当前项目的结论：

```text
1. 该摄像头支持 UVC，适合先用 OpenCV / V4L2 直接读取。
2. 在 PC 和 RK3588 上优先使用 /dev/video0 或实际识别出的 /dev/videoX。
3. 视觉识别第一阶段建议使用 640×480 或 1280×720，降低处理压力。
4. 若只做药盒/标签/二维码识别，建议先使用 MJPEG 640×480 或 MJPEG 1280×720。
5. 若后续做低延迟处理，可测试 YUYV 格式，但带宽和 CPU 压力更高。
6. Docker 部署到 RK3588 时需要挂载摄像头设备，例如 --device=/dev/video0:/dev/video0。
```

## 7. 目标业务流程规划

当前项目建议先明确成一个“按批次配送”的送药系统，而不是只做“单次送药到某个病房”。

目标效果：

```text
药房创建一个配送批次
-> 摄像头扫码/识别药品信息
-> Web 生成本次配送任务列表
-> 按病房 A/B/C 分组
-> 每个病房下包含 2-3 个病人/床位
-> 每个病人下包含 1-N 个药品条目
-> 药房完成装药确认
-> 小车按路线依次到 A/B/C 病房
-> 到每个病房后完成病房确认、病人确认、药品交付确认
-> 全部完成后自动或手动返回药房
-> 生成本次配送记录和异常清单
```

### 7.1 推荐业务对象

建议把业务数据分成四层：

```text
配送批次 DeliveryBatch
  └── 病房站点 DeliveryStop
        └── 病人订单 PatientOrder
              └── 药品条目 MedicationItem
```

字段建议：

```text
DeliveryBatch:
  batch_id
  source_station=pharmacy
  route_status
  created_time
  started_time
  finished_time
  operator_id

DeliveryStop:
  stop_id
  target_station=ward_a/ward_b/ward_c
  display_name=A病房/B病房/C病房
  sequence_index
  stop_status
  arrived_time
  completed_time

PatientOrder:
  order_id
  patient_id
  patient_name
  ward_id
  bed_no
  patient_status
  priority

MedicationItem:
  medication_id
  medicine_name
  product_code
  product_model
  quantity
  trace_id
  order_no
  slot_id
  loaded
  dispensed
  returned
```

当前系统已有 `CreateDeliveryTask`、`DeliveryState`、`VerifyDeliveryTask` 和 `medications_json`，第一版可以先不大改接口，先用 `medications_json` 承载多药清单；后续再正式扩展为批次、站点、病人和药品四层结构。

### 7.2 药房端装药流程

药房端建议这样操作：

```text
1. 药师在 Web 创建“配送批次”
2. 选择本次要送的病房，例如 A、B、C
3. 为每个病房录入病人/床位和药品清单
4. 摄像头扫描药品标签、二维码或条码
5. 系统提取 product_code、product_model、quantity、trace_id、order_no
6. Web 自动匹配药品属于哪个病人订单
7. 药师确认装入对应药仓/药盒/格口
8. 每个药品条目变为 loaded=true
9. 全部药品装药确认通过后，批次进入 READY_TO_DEPART
```

建议 Web 上显示：

```text
本次配送批次：20260514-001
路线：药房 -> A病房 -> B病房 -> C病房 -> 药房

A病房  3个病人  7件药品  已装 7/7
  床位 A-01  张三  已装 2/2  待送达
  床位 A-02  李四  已装 3/3  待送达
  床位 A-03  王五  已装 2/2  待送达

C病房  2个病人  4件药品  已装 4/4
```

药房端必须拦截的问题：

```text
药品扫码不在当前批次中
product_code 不一致
trace_id 不一致
数量不一致
重复装药
未装完就尝试出发
药仓/格口冲突
```

### 7.3 小车配送主流程

正式运行时，小车主流程建议是：

```text
IDLE
-> PREPARING_BATCH
-> SCANNING_MEDICINES
-> WAITING_LOAD_CONFIRMATION
-> READY_TO_DEPART
-> NAVIGATING_TO_WARD
-> ARRIVED_AT_WARD
-> WARD_HANDOVER
-> NAVIGATING_TO_NEXT_WARD
-> RETURNING_TO_PHARMACY
-> BATCH_COMPLETED
-> IDLE
```

以 A、B、C 三个病房为例：

```text
药房装药完成
-> 小车出发去 A 病房
-> 到达 A 病房
-> 护士或药师在 Web/扫码确认“当前是 A 病房”
-> Web 显示 A 病房需要交付的病人列表
-> 逐个病人完成身份确认和药品交付确认
-> A 病房所有病人完成后，A 病房状态变为 completed
-> 小车继续去 B 病房
-> B 病房完成后继续去 C 病房
-> C 病房完成后返回药房
-> 药房确认任务结束
-> 系统生成配送记录
```

### 7.4 到达病房后应该做什么

到达病房后，不建议直接把整个病房任务标记完成，而是进入“交付确认”阶段。

每个病房的交付步骤：

```text
1. 播报：已到达 A 病房，请确认取药
2. Web 显示 A 病房病人列表
3. 护士确认当前病房，可通过病房二维码、按钮或账号确认
4. Web 依次展示当前病人和床位
5. 扫描病人腕带/床头码，或由护士手动确认病人身份
6. 扫描药品码，确认药品属于该病人
7. 点击“确认交付”
8. 系统记录交付时间、扫码值、操作者和结果
9. 该病人所有药品交付完成后，进入下一个病人
10. 当前病房所有病人完成后，允许继续下一病房
```

交付时 Web 应该突出显示：

```text
当前站点：A病房
当前病人：A-01床 张三
应交付药品：
  1. FXL0530-100-M  数量 1  trace_id=202011444  待扫码
  2. XXX            数量 2  trace_id=202011445  待扫码

操作按钮：
  扫码确认药品
  确认交付
  病人暂不在
  药品异常
  暂停任务
  呼叫人工处理
```

### 7.5 异常流程

必须提前设计异常，不然后面实车联调会很混乱。

建议支持这些异常状态：

```text
PAUSED                  人工暂停
BLOCKED                 导航受阻
NEED_MANUAL_ASSISTANCE  需要人工处理
CANCELED                批次取消
ERROR                   系统错误
RETURNING_TO_PHARMACY   带异常返回药房
```

典型异常处理：

```text
药品扫码不匹配：
  不允许交付
  播报请核对药品
  Web 记录 mismatch

病人不在病房：
  标记该病人为 skipped 或 pending_retry
  药品保持未交付
  可以继续本病房下一个病人
  回药房后列入未交付清单

某病房暂时无法进入：
  当前病房标记 blocked
  可以选择跳过该病房先送下一病房
  最后再重试或返回药房

导航失败：
  停车
  播报需要人工协助
  Web 显示当前位置、目标点和错误原因

低电量：
  禁止开始新批次
  运行中优先返回药房或充电点

急停触发：
  立即停止运动
  任务进入 PAUSED
  需要人工解除后才能继续
```

### 7.6 回药房与任务闭环

全部病房处理完之后，不建议直接结束，而是先回药房做闭环确认。

回药房后应做：

```text
1. 播报：本次配送已返回药房
2. Web 显示本次配送结果
3. 列出已完成交付药品
4. 列出未交付/跳过/异常药品
5. 药师确认回收未交付药品
6. 扫码确认剩余药品 trace_id
7. 点击“结束批次”
8. 系统生成审计记录
```

最终报告建议包含：

```text
批次号
路线
开始时间
结束时间
总病房数
总病人数
总药品数
已交付数量
未交付数量
异常数量
每次扫码记录
每次人工确认记录
每个站点到达/离开时间
```

### 7.7 Web 页面建议布局

建议 Web Dashboard 从“单任务状态面板”升级为“配送批次控制台”。

页面区域：

```text

左侧批次列表：
```

### 7.8 和导航系统的关系

业务系统不应该直接控制电机，而是只发“去哪个站点”的目标。

推荐分层：

```text
业务层：
  当前批次、病房顺序、病人清单、药品交付确认

导航任务层：
  接收目标站点，例如 ward_a
  调用 Nav2 NavigateToPose
  返回导航成功/失败/取消

底盘控制层：
  接收 /cmd_vel
  发布 /odom
  处理急停和超时停车
```

在下位机不可用时，可以先实现：

```text
业务流程完整跑通
导航结果用 mock_nav_success 模拟
到站后仍然执行真实扫码确认
Web 能完整显示批次、病房、病人和药品状态
```

等底盘恢复后，再把 `mock_nav_success` 替换为真实 Nav2 action。

## 11. 当前风险点

### 11.1 ROS2 版本差异

PC 端是 Jazzy，RK3588 是 Humble。

风险：

```text
Python 版本不同
部分依赖包版本不同
部分 launch 或第三方包需要适配
```

应对：

```text
尽早在 RK3588 Docker Humble 中构建验证
避免使用 Python 3.12 独有特性
保持接口和 launch 写法兼容 Humble
```

### 11.2 摄像头和 AI 推理性能

风险：

```text
PC 上能跑的 YOLOv8 在 RK3588 CPU 上可能帧率较低
摄像头格式和驱动可能不同
Docker 内访问摄像头需要设备挂载和权限配置
```

应对：

```text
先做图片/摄像头基础识别
先确认 /dev/video0 可用
后续再做 ONNX/RKNN/NPU 优化
```

### 11.3 AIKit SDK 架构

风险：

```text
x86_64 libaikit.so 不能在 RK3588 上运行
```

应对：

```text
基础镜像先跳过 wheeltec_aikit_esr
确认 ARM64 SDK 后再集成
语音主线先使用 M2 串口桥接或模拟命令
```

### 11.4 真实导航复杂度

风险：

```text
真实导航依赖雷达、底盘、IMU、TF、odom、Nav2 多模块协同
调试复杂度高
```

应对：

```text
当前 task_manager 继续保留模拟导航模式
等雷达和底盘稳定后再接 Nav2 action
```
